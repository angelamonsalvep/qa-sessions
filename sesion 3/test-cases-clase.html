Módulo 2 · Concepto

¿Qué es un Test Case?

Un conjunto estructurado de condiciones, pasos e información para verificar si un sistema se comporta como se espera.

❌ Idea vaga

"Probar que el login funciona"

¿Cómo lo pruebo?
¿Con qué datos?
¿Qué se espera exactamente?
¿Pasó o falló?

No reproducible · No verificable · No sirve
✅ Test Case real

Título: Login con credenciales válidas

Precond: Usuario registrado, sin sesión activa

Pasos:

  1. Ir a /login

  2. Ingresar email: test@qa.com

  3. Clic en "Iniciar sesión"

Esperado: Redirige a /dashboard

Cualquier persona del equipo puede ejecutarlo
📋
Documentado

Escrito en formato estándar, no en la cabeza del tester.

🔁
Reproducible

Cualquier persona puede repetirlo con el mismo resultado.

✔️
Verificable

Hay un criterio claro: PASS o FAIL. Sin interpretación.

Regla de oro: Si un colega que nunca vio el sistema ejecuta tu TC, debe obtener exactamente el mismo resultado que tú.

Módulo 3 · Anatomía

Estructura de un Test Case

Cada campo tiene un propósito. Saltarse uno rompe la reproducibilidad.

TC-ID
Identificador único
▾

Un código alfanumérico único. Permite referenciar el TC en Jira, reportes de bugs y TestRail.

TC-LOGIN-001  ·  AUTH-TC-042  ·  CHECKOUT-007

Convención: TC-[MÓDULO]-[NÚMERO] — mantenlo consistente en todo el proyecto.

Título
Descripción corta del objetivo
▾

Responde: ¿Qué se prueba bajo qué condición?

❌ Vago

"Probar login"

✅ Correcto

"Login exitoso con email y contraseña válidos"

Precondiciones
¿Qué debe existir antes de ejecutar?
▾

Estado del sistema, datos de prueba, usuarios, configuración requerida.

• El usuario test@qa.com está registrado y activo
• Contraseña del usuario: "QA2024!"
• Sistema disponible en staging.miapp.com
• El usuario NO tiene sesión activa en el navegador
Pasos
Acciones numeradas, una por línea
▾

Cada paso = una sola acción. Usa datos exactos. Nada ambiguo.

1. Abrir Chrome y navegar a https://staging.miapp.com/login
2. En el campo "Email" ingresar test@qa.com
3. En el campo "Contraseña" ingresar QA2024!
4. Hacer clic en el botón "Iniciar sesión"
5. Observar la respuesta del sistema
Resultado Esperado
El comportamiento correcto del sistema
▾

Concreto, visible y verificable. ¿Qué debe ocurrir exactamente?

❌ Vago

"El usuario entra al sistema"

✅ Concreto

"Redirige a /dashboard y muestra 'Bienvenida, Ana López' en menos de 2s"

Módulo 5 · Anti-patrones

Errores Comunes

Los TCs mal escritos generan discusiones, no resultados. Reconócelos y evítalos.

Error #1
Falta de claridad
Paso 3: "Ingresar datos del usuario"

¿Cuáles datos? ¿Email? ¿Username? Cada tester lo interpreta diferente.

✅ Fix: "Ingresar ana@test.com en el campo Email"

Error #2
Pasos ambiguos
"Hacer las verificaciones necesarias"

¿Qué verificaciones? ¿Cuántas? Esto no es un paso, es una sugerencia.

✅ Fix: Listar cada verificación como paso numerado

Error #3
Resultado no verificable
Esperado: "El sistema funciona bien"

¿Qué es "bien"? No hay criterio objetivo para decidir PASS o FAIL.

✅ Fix: Describir exactamente qué elemento visual aparece

Error #4
Múltiples objetivos
TC-001: "Probar login, registro y recuperar contraseña"

Si falla, ¿cuál parte falló? Un TC = un objetivo. Siempre.

✅ Fix: TC-001, TC-002, TC-003 separados

Regla diagnóstica: Si dos testers ejecutan el mismo TC y obtienen resultados distintos, el TC está mal escrito — no los testers.

Módulo 6 · Best Practices

Buenas Prácticas

Las reglas que definen a un QA Engineer senior. Simples, pero no siempre aplicadas.

📝
Claridad total

Cualquier persona del equipo, incluso alguien nuevo, debe poder ejecutar el TC sin preguntar nada. Datos concretos, no genéricos.

🎯
Un objetivo por caso

Cada TC prueba una sola cosa. Si falla, sabes exactamente qué está roto. TC-001 = login exitoso. TC-002 = login fallido. Nunca mezcles.

🔁
Reproducibilidad garantizada

Precondiciones claras para dejar el sistema en el mismo estado inicial siempre. Usa datos de prueba controlados, nunca datos de producción.

🔍
Especificidad en el resultado

No "muestra error", sino "muestra el mensaje 'Contraseña incorrecta' en rojo debajo del campo, sin redirigir". Textos exactos, comportamiento exacto.

📂
Nomenclatura consistente

Usa siempre el mismo esquema de IDs: TC-[MÓDULO]-[NÚMERO]. Facilita el tracking, los reportes y la comunicación con desarrollo.
